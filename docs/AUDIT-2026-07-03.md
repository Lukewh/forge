## ✅ Issue #4: Linear Sync Broken for All Watcher-Completed Issues

**Fixed** — commit `3c41909`

### Root cause
`agent-runner.js` has a `transition()` wrapper that calls `syncLinearState(newState)`. **`watcher.js` had its own separate `transition()` with no Linear sync.** All watcher-driven transitions (`DONE`, `IN_MERGE_QUEUE`, `AWAITING_FIX_APPROVAL`) were silently skipped. Additionally `IN_MERGE_QUEUE` was absent from `agent-runner.js`'s `LINEAR_STATE_MAP`.

### Evidence
52 of 53 completed issues had `linear_state = 'In Review'` despite `state = 'DONE'`.

### Fix
- Added `syncLinearState()` to `watcher.js` with its own `LINEAR_STATE_MAP` covering `DONE → "Done"`, `IN_MERGE_QUEUE → "In Review"`, `AWAITING_FIX_APPROVAL → "In Review"`, etc.
- Added `IN_MERGE_QUEUE` to `agent-runner.js`'s `LINEAR_STATE_MAP`.

## ✅ Issue #5: `ai_review_rounds` Never Preserves Historical Data

**Fixed** — commit `3c41909`

### Root cause
`ai_review_rounds` was a per-cycle counter that reset to `0` on every approval. After any completed issue, the round history was permanently lost.

### Evidence
`SELECT DISTINCT ai_review_rounds FROM issues` returned only `0` across all 62 issues, despite BAND-5474 needing 5 reviewer runs, BAND-5185 needing 6, BAND-4940 needing 9.

### Fix
- Added `total_ai_review_rounds` column — cumulative, never resets.
- Both columns incremented on each rejection; only `ai_review_rounds` resets on cycle completion.
- Backfilled 34 issues from `activity_log`.

## ✅ Issue #6: Watcher Silently Loops on Bad `wt_path`

**Fixed** — commit `757180b`

### Root cause
When `wt_path` didn't exist on disk, `watcher.js` logged an error, exited code 1, and unlocked the issue without transitioning to `FAILED`. The scheduler respawned it every tick indefinitely (81 consecutive failures on BAND-5422, 32 on BAND-4577).

### Fix
- `wt_path = NULL` → transition immediately to `FAILED`.
- `wt_path` set but dir missing → fall back to `FORGE_DIR` as `cwd` for all `gh` commands (which don't need a git repo dir) and continue watching. Write a `wt_missing_warning` to `activity_log` once per hour so the dashboard surfaces it without spam.

## ✅ Issue #7: Zombie Agent Runs Get Wrong `exited_at` Timestamp

**Fixed** — commit `f23e003`

### Root cause
Dashboard startup cleanup and the reset action both wrote `exited_at = datetime('now')` for orphaned runs. If the dashboard had been offline for days, every zombie run received an `exited_at` far in the future relative to when it actually died — inflating reviewer avg from ~300s to 10,206s.

### Fix
Changed both sites to `datetime(started_at, '+10 minutes')` — the stale-lock reaper threshold, the tightest upper bound on a zombie process's actual lifetime. Backfilled 20 historical rows.

**After fix:** reviewer avg 10,206s → 237s. Watcher max 420,446s → 1,062s.

## ✅ Issue #8: git-agent and fixer Have No Auto-Retry

**Fixed** — commit `f12577f`

### Root cause
Any non-zero exit from `git-agent` or `fixer` immediately transitioned the issue to `FAILED`, requiring manual intervention. 6 git-agent failures and 5 fixer failures were observed, accounting for the majority of the 31 manual advances.

### Fix
- Added `retry_count` column to `issues` and `auto_retry_max` setting (default 3).
- On non-zero exit for `git-agent` or `fixer`: if `retry_count < max`, increment and unlock without transitioning to FAILED — scheduler retries next tick. On max retries exhausted, go FAILED and reset counter.
- Successful exit resets `retry_count = 0`. Dashboard manual retry also resets it.
- LLM agents excluded — their failures are less transient.

## ✅ Issue #9: Verdict Files Deleted After Use

**Fixed** — commit `035e393`

### Root cause
`plan-review-verdict.json` and `review-verdict.json` were deleted immediately after being read via `fs.unlinkSync`, leaving no way to audit whether the AI wrote a real verdict vs. agent-runner defaulting to "approved" due to a missing/crashed run.

### Fix
- Archive verdict before deleting: `fs.renameSync` to `projects/{id}/verdicts/plan-review-round-{N}.json` (or `code-review-round-{N}.json`), where N = current round number.
- When verdict file is missing entirely: write a `missing_verdict` event to `activity_log` (previously only logged to console) so the dashboard surfaces it as a visible signal.
- `verdicts/` dir is untouched by the existing `assets/` cleanup at DONE.
